# Commit Convention

> 프로젝트 커밋 메시지 작성 규칙. 기본적으로 Conventional Commit Guide를 따르며, 모든 AI 에이전트가 공통으로 준수한다.

---

## 형식

```
{type}: {한글 설명}
```

| 구성요소 | 필수 | 설명 |
|----------|------|------|
| `{type}:` | ✅ | 커밋 타입 (아래 표 참조) |
| `{한글 설명}` | ✅ | 한글 자연어, 간결하게. 마침표 없음 |

## 타입

### 기능

| 타입 | 사용 상황 | 예시 |
|------|----------|------|
| `feat` | 기능 구현, 화면/API/서비스 추가 | `feat: 게시글 작성 화면 구현` |
| `init` | 프로젝트/모듈 최초 생성 | `init: Vue 프로젝트 초기화` |

### 유지보수

| 타입 | 사용 상황 | 예시 |
|------|----------|------|
| `chore` | 설정, 의존성, 빌드, 환경 구성 | `chore: agent 문서 최초 등록` |
| `chore(deps)` | 의존성 추가/변경 | `chore(deps): @lucide/vue 의존성 추가` |
| `refactor` | 동작 변경 없는 코드 정리 | `refactor: 패키지명 리팩토링` |
| `refactor({domain})` | 도메인 한정 리팩토링 | `refactor(post): 핸들러 계층 분리` |
| `clean` | 불필요 파일/클래스 제거 | `clean: 미사용 import 제거` |
| `docs` | 문서 작성/수정 | `docs: API 명세 업데이트` |
| `style` | 포매팅, 린트 | `style: 린트 규칙 적용` |

### 버그 및 검증

| 타입 | 사용 상황 | 예시 |
|------|----------|------|
| `fix` | 버그 수정 | `fix: 댓글 작성 시 post_id 누락 문제 수정` |
| `test` | 테스트 작성/수정 | `test: PostService 단위 테스트 작성` |

## 제목 작성 규칙

### 기본 원칙

1. **한글**: 모든 제목은 한글로 작성 (예외: 기술 용어, 클래스명)
2. **50자 제한**: 제목은 50자를 초과할 수 없다. 이를 초과하면 제목은 간결하게, 상세 내용은 본문에 작성한다. 간결한 변경은 제목에 의도를 전부 담을수 있도록 작성한다.
3. **명사형 종결**: "구현", "추가", "수정", "작성", "제거", "분리", "마이그레이션", "교체" 등
4. **마침표 없음**: 제목 끝에 `.` 사용하지 않음

### 본문 작성 규칙

- 본문은 **반드시 불릿 리스트**로 구조화한다. 한 문단으로 나열하는 것은 금지한다.
- 각 불릿은 하나의 변경 단위만 담는다.
- 신규 파일은 `(new)`, 삭제된 파일은 `(removed)`, 기능 중단(deprecated)은 `(deprecated)` 상태를 파일명 뒤에 표기한다. 수정된 파일은 별도 상태를 표기하지 않는다.
- 형식:
  ```
  - {파일명 또는 변경 요약} (new|removed|deprecated) — 부가 설명
  ```

### 권장 동사

| 패턴 | 예시 |
|------|------|
| `XX 구현` | `게시글 작성 화면 구현` |
| `XX 추가` | `게시글 CRUD API 추가` |
| `XX 수정` | `빌드 오류 수정` |
| `XX 구성` | `프론트엔드 CI 파이프라인 구성` |
| `XX 작성` | `BaseEntity 작성` |
| `XX 제거` | `사용하지 않는 컴포넌트 제거` |
| `XX 분리` | `HTML 템플릿 별도 파일로 분리` |

### API 엔드포인트 포함 시

API 관련 커밋은 제목 끝에 HTTP 메서드와 경로를 괄호로 표기:

```
feat: 게시글 수정 API 구현 (PUT /api/posts/{id})
feat: 게시글 목록 조회 API 구현 (GET /api/posts)
```

## 컴포넌트 커밋 순서

작업은 논리적 컴포넌트(계층) 단위로 분할하여 커밋한다. 커밋 순서는 **의존성의 역순**을 따른다:

```
데이터 계층(Entity/Repository) → 비즈니스 로직 계층(Service) → 정제 계층(Controller/API)
  → 클라이언트 API 연동 → 상태 관리 계층(Store/Service) → 화면(View/UI)
```

예시:
1. `feat: 공통 응답 DTO 및 Entity/Repository 추가`
2. `feat: Service 계층 비즈니스 로직 및 조건 처리 구현`
3. `feat: Controller/API 계층 엔드포인트 및 입력 검증 적용`
4. `feat: 클라이언트 응답 타입 정의 및 공통 컴포넌트 확장`
5. `feat: 클라이언트 API/Service/Store 계층 연동`
6. `feat: 화면 View UI 및 인터랙션 구현`

- 각 컴포넌트는 하나의 계층 또는 하나의 관심사만 포함한다.
- 설정 파일 및 지침 문서(예: `CLAUDE.md`, `agent/`) 변경은 코드 커밋과 분리하여 `docs:` 타입으로 별도 커밋한다.

## 커밋 전 확인 사항

- **`fix` 타입 커밋**: 버그 수정은 커밋 직전에 즉시 빌드 확인한다.
  - 프로젝트 빌드/컴파일 명령어는 현재 도메인의 빌드 도구를 사용한다 (tech-specific 커맨드는 `agent/project/` 문서를 참조).
- **`feat`, `refactor`, 그 외 타입**: 개별 컴포넌트 커밋 단계에서는 빌드 정합성을 확인하지 않는다. 빌드 확인은 작업 완료 직전 1회만 수행한다.
- 빌드 실패 상태로 최종 커밋해서는 안 된다.
